Overview

The ONX scaffold is a single-file HTML template that gives every presales prototype a consistent look and feel. It encodes the OneNexus brand, layout system, and AI chat interface so that each demo communicates the same product quality — regardless of which industry vertical or client it's built for.

Single file
Zero build step. Open in browser, share as email attachment, deploy as static file.
Agent-ready
AI chat panel is a first-class citizen — not bolted on. Every interaction pattern is already built.
Customisable
Swap the app name, nav items, and content pages. The strict brand shell stays intact automatically.
Strict patterns must never be changed for any client prototype. They are the visual contract that makes every ONX demo instantly recognisable.
Flexible patterns should be adapted per client. App name, sidebar pages, content data, and domain vocabulary are all yours to change.

Strict

The AI avatar communicates state through glow intensity. Three states exist: idle (subtle, always-on), thinking (bright fast pulse while generating), and hero (warm 3s for the welcome screen). CSS handles all three — JS only toggles a class.

Panel switcher JS 
function showSettingsPanel(panelId, navEl) {
  // Deactivate all nav items
  document.querySelectorAll('#page-settings .settings-nav-item')
    .forEach(el => el.classList.remove('is-active'));
  if (navEl) navEl.classList.add('is-active');

  // Show matching panel, hide all others
  document.querySelectorAll('#page-settings .settings-panel').forEach(p => {
    p.hidden = (panelId == null) ? true : (p.dataset.panel !== panelId);
  });
}

New App Workflow

The goal is to build a OneNexus-branded demo app for client presentations. The workflow is agent-assisted via Claude Code — you provide a spec file, the agent composes pages from scaffold components and builds them one at a time for your review.

Before you start — have these ready
Required App name — shown in the browser tab and sidebar (e.g. "Meridian Advisor Hub")
Required Client spec file — markdown doc, demo HTML, or JSX. A description typed in chat is not sufficient.
Recommended User persona — name and initials for the demo user (e.g. "Sarah Chen / SC")
Recommended Key interactions — what happens when a user clicks a row, card, or button
1
Open a Claude Code session and say "onx scaffold"
The agent will confirm your inputs and ask for the spec file before doing anything. It will not build from a verbal brief.
2
Review and approve the Build Plan
The agent reads the spec and proposes a page list, component choices, and every identified interaction (drill-downs, modals, filters). No HTML is written until you approve.
3
Review each page before the next is built
Pages are built one at a time. The agent shares the file after each page and waits for explicit approval — say "approved" or give feedback. It will not continue until you do.
4
Receive the delivered file
On completion the agent removes the Components reference page, runs the locked checklist, and saves to apps/[AppName]-app.html. The chat panel, brand tokens, and avatar markup are untouched throughout.
Manual fallback (no Claude Code)
  1. Copy template/onx-app-scaffold.html → rename [AppName]-app.html
  2. Replace the two [App Name] occurrences: <title> and .app-name span. Do not touch the logo <img>
  3. Edit typewriterMessages array — 3–4 domain-specific prompts, under 50 chars each
  4. Update id="welcome-name" and avatar initials in sidebar footer and chat bubbles
  5. Add custom pages using the NEW PAGE TEMPLATE at the file bottom. Pick components from page-components
  6. Fill the 3 DEMO anchors: user identity, thread list, user profile (settings)
  7. Remove the nav-components nav item before delivering

Abstracting an Existing App

When a client has an existing application, the scaffold becomes the presentation layer. The goal is to make the client's logic feel native to the ONX experience without rebuilding it.

1
Map screens to scaffold pages
Identify which existing app screens correspond to scaffold sections — e.g. existing reports → Runs/Review page, existing settings → Settings page.
2
Keep the entire chat panel intact
Never modify .flat-panel, .chat-turn, or avatar markup. The chat panel is the ONX differentiator — it must be pixel-identical across all prototypes.
3
Replace DEMO content blocks
Content between <!-- DEMO --> and <!-- END DEMO --> tags is yours to swap. Use real or representative client data in these blocks.
4
One nav layer only
If the existing app has its own navigation, map items 1:1 to .nav-item entries. Do not add a second nav layer — it breaks the canvas layout and the sidebar width assumption.
