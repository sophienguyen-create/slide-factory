Overview

The ONX scaffold is a single-file HTML template that gives every presales prototype a consistent look and feel. It encodes the OneNexus brand, layout system, and AI chat interface so that each demo communicates the same product quality — regardless of which industry vertical or client it's built for.

Single file
Zero build step. Open in browser, share as email attachment, deploy as static file.
Agent-ready
AI chat panel is a first-class citizen — not bolted on. Every interaction pattern is already built.
Customisable
Swap the app name, nav items, and content pages. The strict brand shell stays intact automatically.
Strict patterns must never be changed for any client prototype. They are the visual contract that makes every ONX demo instantly recognisable.
Flexible patterns should be adapted per client. App name, sidebar pages, content data, and domain vocabulary are all yours to change.

Tool Call Block

Strict

Renders a single agent tool / system call — a query, an MCP call, an extraction step — as a distinct, machine-flavored line. Recessed fill with a left accent rail signals "this is tooling, not prose." Static output record: no hover or interactive state. The tag label is configurable (TOOL, MCP, SQL, …) but always short and uppercase.

Activity Trace

Strict

Collapses a run of intermediate agent steps (tool calls, system calls, reasoning lines) into one compact, indented, expandable cluster — so a long agent process reads as a clean spine of milestones with the detail tucked away on demand. The live (streaming) group auto-expands; it auto-collapses to N steps · <last step> on completion. Clicking toggles, and a manual choice sticks.

COLLAPSED — click to expand
Parsing incoming request from Sarah Smith…
TOOLIntent: tender offer eligibility
SQLSELECT * FROM holdings WHERE record_date <= cutoff
Applying eligibility rules across 142 records…
MCPCalling Security Master via MCP gateway
LIVE — auto-expands while streaming, auto-collapses on completion
View markup + JS helpers 
// Static — render collapsed or expanded from an array of trace lines:
traceGroupMarkup({
  steps: [
    'Parsing incoming request…',
    toolCallMarkup({ tag: 'TOOL', body: 'Intent: tender offer eligibility' }),
    toolCallMarkup({ tag: 'MCP', body: 'Calling Security Master via MCP gateway' }),
  ],
  open: false,   // collapsed → "3 steps · Calling Security Master…"
});

// Live — stream steps into the latest group, then collapse on completion:
startTrace(messagesContainer);                 // appends an expanded live group
addTraceStep('Parsing incoming request…');     // each call updates count + peek
addTraceStep(toolCallMarkup({ tag: 'TOOL', body: 'Intent: tender offer eligibility' }));
completeTrace();                               // auto-collapses (unless user toggled)

// Manual toggle is wired via onclick="toggleTrace(this)" in the markup.
Toggle is a real <button> with aria-expanded. Chevron rotates 90° on open; the list sits behind a 1px --border-subtle rail, indented ~12px past the spine. No per-line timestamps or copy actions inside a trace — it's a detail view.
Behaviour is strict: live group expanded, auto-collapse on completion, manual override sticks. Drive it with startTrace / addTraceStep / completeTrace.
Flexible: the step content — reasoning lines and the tags/bodies of any nested tool-call blocks.

function showSettingsPanel(panelId, navEl) {
  // Deactivate all nav items

New App Workflow

The goal is to build a OneNexus-branded demo app for client presentations. The workflow is agent-assisted — you provide a spec file, the agent composes pages from scaffold components and builds them one at a time for your review. Works with Claude Code, Codex CLI, or Cursor.

Before you start — have these ready
Required App name — shown in the browser tab and sidebar (e.g. "Meridian Advisor Hub")
Required Client spec file — markdown doc, demo HTML, or JSX. A description typed in chat is not sufficient.
Recommended User persona — name and initials for the demo user (e.g. "Sarah Chen / SC")
Recommended Key interactions — what happens when a user clicks a row, card, or button
1
Clone the repo
git clone https://github.com/Innovatube/onx-scaffold, then cd onx-scaffold. This gives you the scaffold template, component library, and agent instructions locally.
2
Install the pre-commit hook (one-time setup)
Run cp scripts/pre-commit .git/hooks/pre-commit && chmod +x .git/hooks/pre-commit. This keeps the scaffold and design-system versions in sync on every commit — do it once per clone.
3
Open the project folder in your AI IDE
Claude Code, Codex CLI, or Cursor all work. They pick up AGENTS.md and CLAUDE.md automatically, so the agent already knows the scaffold rules before you type anything.
4
Drop the client spec file into specs/
Accepted formats: markdown, demo HTML, or JSX. A description typed in chat is not sufficient — the agent will ask for a file if none is present.
5
Type "onx scaffold" and name the spec file
Say which file to use — e.g. specs/my-client-spec.html. There may be multiple files in specs/; the agent will not guess. It will not build from a verbal brief.
6
Review and approve the Build Plan
The agent reads the spec and proposes a page list, component choices, and every identified interaction (drill-downs, modals, filters). No HTML is written until you approve.
7
Review each page before the next is built
Pages are built one at a time. The agent shares the file after each page and waits for explicit approval — say "approved" or give feedback. It will not continue until you do.
8
Receive the delivered file
On completion the agent inlines core.css/core.js into one self-contained file, runs the locked checklist, and saves to apps/[AppName]-app.html. The chat panel, brand tokens, and avatar markup are untouched throughout.
Manual fallback (no AI IDE)
  1. Start from template/layout-shell.html → rename [AppName]-app.html
  2. Inline core.css into <style> and core.js into <script> (replace the two linked tags) so the file is self-contained
  3. Replace the two [App Name] occurrences: <title> and .app-name span. Do not touch the logo <img>
  4. Set window.ONX_TYPEWRITER in an inline script before core.js — 3–4 domain prompts, under 50 chars
  5. Update id="welcome-name" and avatar initials in sidebar footer and chat bubbles
  6. Add custom pages using the NEW PAGE TEMPLATE at the file bottom. Pick components from this design system
  7. Fill the 3 DEMO anchors: user identity, thread list, user profile (settings)

Abstracting an Existing App

When a client has an existing application, the scaffold becomes the presentation layer. The goal is to make the client's logic feel native to the ONX experience without rebuilding it.

1
Map screens to scaffold pages
Identify which existing app screens correspond to scaffold sections — e.g. existing reports → Runs/Review page, existing settings → Settings page.
2
Keep the entire chat panel intact
Never modify .flat-panel, .chat-turn, or avatar markup. The chat panel is the ONX differentiator — it must be pixel-identical across all prototypes.
3
Replace DEMO content blocks
Content between <!-- DEMO --> and <!-- END DEMO --> tags is yours to swap. Use real or representative client data in these blocks.
4
One nav layer only
If the existing app has its own navigation, map items 1:1 to .nav-item entries. Do not add a second nav layer — it breaks the canvas layout and the sidebar width assumption.
